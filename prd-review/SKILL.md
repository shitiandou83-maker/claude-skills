# PRD 评审自检

在需求评审前，让 AI 扮演不同角色对 PRD 进行多角度挑战，提前暴露问题。

## 核心原则

1. **多角色视角**：从不同利益相关方的角度审视需求，不局限于产品经理单一视角
2. **聚焦风险**：找问题、找漏洞、找矛盾，而不是找"写得好"的地方
3. **可操作反馈**：每个问题附带具体建议，不说"这里有问题"而说"这里应该怎么改"
4. **评审前自检**：不是替代正式评审，而是在正式评审前做一轮"虚拟评审"，让正式评审更聚焦

## 工作流程

### 1. 获取评审材料

根据用户提供的输入类型，读取 PRD 内容：

**本地文档路径**
- 使用 Read 工具读取 PRD 文档（.md 格式）
- 同时读取相关的业务知识库（`产品管理/{产品名称}/业务知识库.md`），了解产品上下文
- 如果有原型文件，读取原型 HTML 了解页面交互

**对话直接输入**
- 用户已在对话中提供需求内容或方案描述，直接进入评审

### 2. 选择评审角色

向用户确认需要哪些角色参与评审：

> "请选择需要参与评审的角色（可多选）：
> - 👨‍💻 研发：技术实现风险、接口依赖、数据一致性、性能瓶颈
> - 🎨 设计：交互复杂度、用户理解成本、信息层级、视觉一致性
> - 📊 运营：配置成本、后续维护、推广难度、培训成本
> - 👤 用户：使用过程中可能困惑的地方、学习成本、操作便捷性
> - 💼 老板：业务价值、投入产出比（ROI）、战略对齐度、竞争壁垒
> - ✅ 全选（所有角色一起评审）"

**默认建议**：至少选择研发 + 老板，因为这是评审会上最常被问到的问题。

### 3. 执行评审

对每个选中的角色，AI 按以下维度逐项检查 PRD：

#### 👨‍💻 研发视角

| 检查维度 | 具体检查项 |
|----------|-----------|
| **技术可行性** | 技术选型是否合理？是否有更简单的实现方式？ |
| **接口依赖** | 需要哪些上下游接口？接口改造影响哪些系统？接口协议是否明确？ |
| **数据一致性** | 数据来源是否可靠？是否存在数据不一致风险？如何保证最终一致性？ |
| **性能风险** | 高并发场景如何处理？大数据量查询是否有保护？是否有缓存策略？ |
| **异常处理** | 异常场景是否覆盖完整？（超时、重试、降级、限流、熔断） |
| **安全风险** | 是否存在数据泄露风险？权限控制是否合理？数据脱敏是否需要？ |
| **可扩展性** | 数据模型是否有扩展性？接口设计是否考虑扩展？ |
| **技术债务** | 是否有过度设计？是否引入了不必要的复杂度？ |

#### 🎨 设计视角

| 检查维度 | 具体检查项 |
|----------|-----------|
| **交互复杂度** | 用户完成任务需要多少步？是否可以减少步骤？是否有更简洁的交互方案？ |
| **用户理解成本** | 功能命名是否直观？操作逻辑是否符合用户心智模型？是否需要额外学习？ |
| **信息层级** | 信息展示是否有主次？关键信息是否突出？是否信息过载？ |
| **操作容错** | 是否支持撤销？误操作后是否有确认？是否有操作引导？ |
| **状态可见性** | 操作结果是否有反馈？加载/处理中是否有提示？错误信息是否友好？ |
| **视觉一致性** | 是否与现有产品风格一致？是否遵循设计规范？ |

#### 📊 运营视角

| 检查维度 | 具体检查项 |
|----------|-----------|
| **配置成本** | 功能上线需要多少配置项？是否可以预置默认值？是否支持批量配置？ |
| **维护成本** | 上线后可能需要多少人工介入？是否需要持续维护？是否有自愈能力？ |
| **推广难度** | 新功能如何让用户知道？是否需要培训？是否有使用指南？ |
| **培训成本** | 用户是否需要培训？培训材料是否需要准备？是否有操作手册？ |
| **数据监控** | 是否需要数据埋点？核心指标是什么？是否有效果评估机制？ |

#### 👤 用户视角

| 检查维度 | 具体检查项 |
|----------|-----------|
| **使用场景** | 在什么场景下使用这个功能？频率如何？是否是刚需？ |
| **操作便捷性** | 是否容易上手？关键操作是否能快速完成？是否支持批量操作？ |
| **学习成本** | 第一次使用需要多少时间理解？是否有引导？文档是否清晰？ |
| **常见困惑** | 哪些地方可能让用户困惑？错误提示是否友好？边界情况是否明确？ |
| **权限限制** | 是否有权限门槛？权限申请流程是否复杂？是否有临时权限方案？ |

#### 💼 老板视角

| 检查维度 | 具体检查项 |
|----------|-----------|
| **业务价值** | 这个需求解决什么业务问题？是否有数据支撑？ROI 如何？ |
| **战略对齐** | 是否和团队/公司战略方向一致？优先级是否合理？ |
| **竞争壁垒** | 是否建立竞争优势？竞品是否已有类似功能？差异化在哪？ |
| **风险预判** | 是否有合规风险？是否有法务问题？是否涉及敏感领域？ |
| **资源评估** | 需要投入多少人力/时间？是否有更高效的替代方案？是否值得投入？ |

### 4. 输出评审报告

按以下结构输出 Markdown 评审报告：

```markdown
# PRD 评审自检报告

**评审对象**：[PRD 文档名称]
**评审日期**：YYYY-MM-DD
**评审方式**：AI 虚拟评审
**参与角色**：[选中的角色列表]

---

## 一、总体评价

[一句话总结：这个 PRD 质量如何，主要风险在哪]

## 二、分角色评审

### 👨‍💻 研发视角
> 共发现 X 个问题

1. **【技术可行性】** [问题描述]
   - 建议：[具体建议]
   - 风险等级：🔴高 / 🟡中 / 🟢低

2. **【接口依赖】** [问题描述]
   - 建议：[具体建议]
   - 风险等级：🔴高 / 🟡中 / 🟢低

### 🎨 设计视角
...

### 📊 运营视角
...

### 👤 用户视角
...

### 💼 老板视角
...

## 三、高风险问题汇总
| 序号 | 问题 | 风险等级 | 影响范围 | 建议 |
|------|------|----------|----------|------|
| 1 | [问题] | 🔴高 | [范围] | [建议] |

## 四、评审总结
- 发现高风险问题 X 个，中风险 X 个，低风险 X 个

```

### 5. 可选：对比历史评审

如果存在上一版本的 PRD 和评审报告，可以对比：

- 上一版本高风险问题是否已修复？
- 新版本是否引入了新的风险？
- 历史决策是否仍然合理？

## 输出检查清单

- [ ] **角色选择**：已确认用户需要哪些角色参与评审
- [ ] **逐角色检查**：每个角色按维度逐项检查，不遗漏
- [ ] **风险分级**：每个问题标注风险等级（高/中/低）
- [ ] **可操作建议**：每个问题附具体改进建议
- [ ] **报告输出**：生成结构化评审报告

## 关键原则

- **多角色视角** — 从不同利益相关方角度审视
- **找问题不找优点** — 聚焦风险、漏洞、矛盾
- **可说可改** — 每个问题附具体建议
- **评审前自检** — 不是替代正式评审，而是降低正式评审的沟通成本
- **历史可追溯** — 如有历史评审报告，对比检查遗留问题是否已修复

### 产出物

评审报告保存至：

```
产品管理/{产品名称}/{版本号}/评审记录/{YYYYMMDD}-{角色列表}.md
```

**文件命名规则**：时间戳 + 角色列表（用 - 连接），如 `20260701-研发-用户-老板.md`。每次评审生成新文件，不覆盖历史记录。

> 📋 **模板**：评审报告按以下结构输出：总体评价 → 分角色评审（每个角色按维度逐项检查）→ 高风险问题汇总 → 评审总结
