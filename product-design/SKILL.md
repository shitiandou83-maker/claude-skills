# 产品设计

## 概述

本技能通过四步流程帮助用户完成完整的产品设计：

1. **竞品分析**（可选）— 分析市场竞品，输出竞品分析报告
2. **需求分析**（必需，不可跳过）— 通过对话式引导，将想法转化为完整的需求文档
3. **产品原型**（可选）— 根据需求文档生成产品原型（HTML+JavaScript）
4. **PRD文档**（可选）— 根据需求文档和原型生成产品需求文档

支持**新需求设计**、**已有产品版本迭代**和**修改已有版本**三种模式，所有产出物按版本号管理。

---

## 准备工作

**在开始任何步骤之前，必须先完成以下准备工作：**

1. **询问产品名称**：向用户询问本次要设计的产品名称（如"数据中心"、"订单管理系统"等）

2. **确认需求类型**：询问用户本次是什么需求类型：
   > "请确认本次需求类型：
   > - 🆕 新需求设计（首次设计该产品）
   > - 🔄 版本迭代（已有产品，在此基础上迭代升级）
   > - ✏️ 修改已有版本（对已有版本号的文件进行小幅度修改）"

3. **确定版本号并创建/确认目录**：
   - **新需求**：版本号默认为 `1.0`，创建目录 `产品管理/{产品名称}/1.0/`
   - **版本迭代**：询问用户输入最新版本号，创建目录 `产品管理/{产品名称}/{版本号}/`
   - **修改已有版本**：询问用户要修改的版本号，必须验证目录 `产品管理/{产品名称}/{版本号}/` 已存在，如不存在则提示用户重新输入
   - 如果产品目录已存在，则不重新创建

4. **确认流程**：向用户说明四步流程，并告知第一步竞品分析为可选步骤
   - **修改已有版本模式特殊说明**：在修改已有版本模式下，所有步骤均可跳过，且需求分析步骤不再是必需的

## 第一步：竞品分析（可选）

### 启动确认

开始前必须询问用户：

> "第一步是竞品分析，这一步为可选步骤。是否需要进行竞品分析？
> - ✅ 是，进行竞品分析
> - ⏭️ 跳过，直接进入需求分析"

如果用户选择跳过，直接进入第二步。

**⚠️ 修改已有版本模式**：询问方式不变，所有步骤均可跳过。

### 竞品分析流程

如果用户选择进行竞品分析，按以下步骤进行：

1. **了解分析目标**
   - 询问用户要分析的产品领域/方向
   - 询问用户是否有已知的竞品名单，或需要自行调研
   - 询问分析的重点维度（功能、用户体验、商业模式、技术架构等）
   - **如果是版本迭代**：引导用户关注新版本与竞品的差异化对比

2. **竞品信息收集**
   - 逐一询问或引导用户确认要分析的竞品（建议3-5个）
   - 对每个竞品，了解以下信息：
     - 产品名称及所属公司
     - 目标用户群体
     - 核心功能
     - 优势与不足
     - 商业模式（如已知）
     - **如果是版本迭代**：竞品的最新变化/更新

3. **对比分析**
   - 引导用户从多个维度对比各竞品
   - 每次只聚焦一个维度的对比
   - 记录用户的分析和判断

4. **总结与建议**
   - 基于竞品分析，提出产品差异化建议
   - 识别市场机会和潜在风险

## 第二步：需求分析（可选）

> **注意**：
> - 新需求和版本迭代模式下，此步骤**必需**，不可跳过。
> - **修改已有版本**模式下，此步骤**可选**，可以跳过。

### 启动确认

开始前必须询问用户：

> "第二步是需求分析，此步骤为「修改已有版本」模式下可选，新需求/版本迭代必选。是否现在开始需求分析？
> - ✅ 是，开始需求分析
> - ⏭️ 跳过，进入下一步"

如果用户选择跳过：
- **修改已有版本模式**：允许，直接进入第三步。
- **新需求或版本迭代模式**：不允许跳过，必须执行需求分析。

### 需求分析流程

**采用 `requirements-analysis` 技能的方法论进行需求分析：**

1. **理解项目背景**
   - 查看当前项目状态（已有文件、文档、竞品分析报告等）
   - 如果第一步已完成竞品分析，参考竞品分析报告中的建议
   - **⚠️ 强制要求（版本迭代）**：如果是版本迭代，在开始需求分析前，必须强制读取业务知识库（`产品管理/{产品名称}/业务知识库.md`）和上一版本的所有文档，深入理解历史设计决策、已定模块结构、关键决策记录和术语定义。在对话中必须明确引用已有知识库的内容来指导分析，避免与历史设计冲突。
   - **⚠️ 修改已有版本模式**：如果是修改已有版本，必须先读取该版本下已有文件（需求文档、原型、PRD 等），了解当前版本的全部内容，然后针对用户要修改的部分进行引导分析。不需要重新走完整的需求分析流程，只聚焦于用户要修改的具体内容。
   - 逐一提问以细化产品想法
   - 尽可能使用选择题，开放式问题也可以
   - 每条消息只提一个问题
   - 聚焦于理解：目的、约束条件、成功标准

2. **探索方法**
   - 提出 2-3 种不同的方法，说明各自的权衡取舍
   - 以对话方式呈现各选项，给出推荐与理由
   - 首先提出推荐的方法，并解释原因

3. **呈现设计**
   - 将设计方案拆分为每节 200-300 字的小节
   - 每节结束后询问用户是否满意
   - 涵盖：架构、组件、数据流、错误处理
   - 如不清楚，随时回头澄清
   - **如果是版本迭代**：明确标注新增、修改、删除的功能

4. **验证确认**
   - 设计方案确认后，整理为完整的需求文档

## 第三步：产品原型（可选）

### 启动确认

开始前必须询问用户：

> "第三步是根据需求文档生成产品原型，此步骤为可选步骤。是否需要生成产品原型？
> - ✅ 是，生成产品原型
> - ⏭️ 跳过，直接进入PRD文档"

如果用户选择跳过，直接进入第四步。

**⚠️ 修改已有版本模式**：询问方式不变，所有步骤均可跳过。如果用户选择生成原型，优先读取该版本已有原型文件，在已有原型基础上修改。

### 原型生成流程

1. **读取需求文档和业务知识库**
   - 读取第二步生成的当前版本需求文档（如有）
   - 提取所有功能模块和页面需求
   - **⚠️ 强制要求（版本迭代）**：如果是版本迭代，必须强制读取业务知识库（`产品管理/{产品名称}/业务知识库.md`），了解已有功能架构和历史原型设计风格。优先展示新增和修改的功能模块，保持与历史版本的设计一致性。
   - **⚠️ 修改已有版本模式**：必须先读取该版本已有的原型文件（`产品管理/{产品名称}/{版本号}/原型.html`），在已有原型基础上进行修改，而非重新生成。保持已有 UI 风格和代码结构一致。

2. **读取 B 端设计规范（强制）**
   - **⚠️ 强制要求**：在生成任何 B 端产品原型前，必须读取并遵循以下三篇规范文档：
     - `knowledge/b端页面设计规范.md`：掌握 B 端通用组件（卡片、表格、弹窗、抽屉等）、文本样式（标题/正文/注释层级）、操作按钮（主要/次要/查询/重置/表格内/扩展按钮）以及导航框架的规范要求
     - `knowledge/b端列表页规范.md`：掌握列表页标准布局（1440×900 设计尺寸）、组件构成（jdesign-react 基础组件 + jdesign-react-pro 高级组件）、框架布局（顶导航48px/左导航200px/页面信息区/内容区）、有/无操作按钮区两种场景的完整布局规则
     - `knowledge/b端图标规范.md`：掌握常用图标命名（Outlined/Filled）及典型使用场景速查
   - **如果产品不属于 B 端类型**（如 C 端 App、H5 活动页、小程序等），可跳过此步骤

3. **确认原型范围**
   - 向用户确认需要生成原型的页面/模块范围
   - 确认风格偏好（如配色、布局风格等）
   - **如果是版本迭代**：确认是否需要继承已有原型的设计风格

4. **生成原型**
   - 使用 HTML + JavaScript 生成可交互的产品原型
   - 原型要求：
     - **B 端场景强制遵守**：严格遵循 B 端设计规范的布局框架、组件使用、间距规则和样式定义
     - 采用现代化 UI 设计风格
     - 使用响应式布局
     - 包含页面导航和路由
     - 核心交互可用（如弹窗、表单、列表筛选等）
     - 使用 Tailwind CSS 或内联样式进行美化
     - 可使用 CDN 引入必要的库（如 Vue.js、Tailwind CSS 等）
     - **版本号标识**：在原型页面顶部或合适位置标注版本号

5. **原型验证**
   - 向用户展示原型效果
   - 根据反馈进行调整

**原型 HTML 技术要求：**

- 单文件 HTML，包含所有 CSS 和 JavaScript
- 使用 Tailwind CSS（通过 CDN）
- 包含以下基础结构：
  - 顶部导航栏（含版本号标识）
  - 侧边栏菜单（根据功能模块生成，标注新增/修改/删除状态）
  - 主内容区域
  - 各功能页面切换逻辑
- 交互效果：
  - 菜单切换
  - 表单交互
  - 弹窗/模态框
  - 列表展示与筛选
  - 数据状态变化反馈

**⚠️ B 端原型必须遵循的规范要点**（来源：`knowledge/` 目录下三篇规范文档）：

- **设计尺寸**：后台系统界面设计尺寸为 1440×900px，背景色 surface-100
- **框架布局**：
  - 顶导航高度 48px，左右 padding 20px，使用 ProTopMenu 组件
  - 左导航宽度 200px，使用 ProSingleMenu 或 ProMenu 组件
  - 页面信息区高度 28px，含面包屑（必有）和扩展操作区（可有）
  - 页面内容区由白色卡片容器包裹，圆角上 8px / 下 0px，左右 padding 24px，上 padding 20px
- **页面标题**：H1 级别，字号 18-20px，字重 Semibold/Medium，色值 text-400
- **数据查询区**：使用三列栅格式表单，label 对齐，查询按钮（primary）+ 重置按钮（plain），超过五项支持展开收起
- **表格容器**：1px border（line-100），圆角 4px，内容截断用"..."，hover 显示 Tooltip
- **状态列颜色**：进行中蓝、失效/完成灰、待处理橙、已通过绿、异常红
- **操作列**：链接按钮（type=link），间距 8px，无图标
- **按钮规范**：
  - 主要操作：type=primary，主题色填充，须有前置图标
  - 次要操作：ghost type=plain，线性样式，无图标
  - 查询按钮：ghost type=primary，主题色填充，无图标
  - 重置按钮：ghost type=plain，线性样式，无图标
  - 扩展操作：ghost type=text，文本样式，可配后置图标
  - 表格内按钮：link type=primary，链接样式，无图标
- **分页**：Pagnation 组件，居右布局，与表格右对齐
- **间距规范**：
  - 页面标题 → 数据查询：20px
  - 数据查询 → 表格容器：24px（无操作按钮区）/ 16px → 分割线 → 16px（有操作按钮区）
  - 表格容器 → 分页器：16px
- **图标使用**：参照 `knowledge/b端图标规范.md` 中的图标命名和场景速查表

## 关键原则

- **渐进式引导** — 每一步开始前必须询问用户确认，不擅自开始
- **一步一确认** — 每个步骤完成后，确认后才进入下一步
- **需求分析可跳过（仅修改模式）** — 修改已有版本模式下，需求分析可选；新需求和版本迭代仍然必需
- **所有步骤可跳过（修改模式）** — 修改已有版本模式下，四步均可跳过
- **其他步骤可跳过** — 新需求/版本迭代模式下，第一步竞品分析、第三步产品原型、第四步PRD文档均可跳过
- **版本化管理** — 所有产出物按版本号目录管理，新需求 V1.0，迭代新版本号
- **修改已有版本** — 在已有版本目录内修改文件，不改变版本号，所有步骤均可跳过
- **知识库沉淀可选（修改模式）** — 修改已有版本模式，知识库更新为可选（询问用户）；新需求和版本迭代自动沉淀
- **文件统一管理** — 所有产出物保存在同一产品版本目录下
- **一次只问一个问题** — 在需求分析阶段，不要用多个问题让人不知所措
- **优先使用选择题** — 在可能的情况下，比开放式问题更容易回答
- **坚决贯彻 YAGNI** — 从所有设计中移除不必要的功能
- **探索替代方案** — 在确定前始终提出 2-3 种方法
- **保持灵活** — 当某些内容不清楚时回头澄清
- **模板严格遵循** — PRD 文档必须严格按照模板结构生成
- **迭代可追溯** — 版本迭代时标注新增/修改/删除，保持历史可追溯
- **B 端规范强制遵循** — 生成 B 端产品原型时，必须读取 `knowledge/` 目录下的三篇规范文档，严格遵循 DongDesign 2.0 的布局框架、组件使用、间距规则、文本样式和按钮规范
