# User Guide Creator

将需求文档转换为面向终端用户的功能使用指引，输出三个文件。

## 核心原则

1. **用户视角**：站在用户角度思考，用"对话感"代替"说明书体"，让用户感受到被服务
2. **价值优先**：功能概述第一句话就要点明"这能帮我省什么事/解决什么痛点"
3. **视觉友好**：善用 emoji 图标、引用块、表格等元素，降低阅读疲劳
4. **服务意识**：遇到限制条件时，提供"解决路径"而非冷冰冰的阻断

## 输出文件

| 文件名 | 格式 | 说明 |
|--------|------|------|
| `用户手册.docx` | Word | 完整用户手册，含原型页面截图（如原型 HTML 存在） |
| `用户手册.md` | Markdown | 完整用户手册，图片位置用占位符标注 |
| `功能简介.md` | Markdown | 独立的功能简介文档，包含模块名称、关键词、适用场景等 |

**三个文件必须放在与 PRD 文档相同的目录下。**

## 工作流程

### 1. 获取需求材料

根据用户提供的输入类型，选择对应的读取方式：

**本地文档路径**
- 使用 Read 工具直接读取文档内容
- 支持格式：.md, .txt, .docx, .pdf 等

**在线文档链接**
- Joyspace 文档：使用 `mcp__JoyspaceParser__ReadJoyspace` 工具
- Relay 设计稿：使用 `mcp__JoyspaceParser__ReadJoyspaceDraft` 工具
- 其他链接：使用 WebFetch 工具

**对话直接输入**
- 用户已在对话中提供需求内容，直接进入分析阶段

如果用户提供多种材料（如同时提供文档和设计稿），全部读取后综合分析。

### 2. 分析需求内容

从需求材料中提取以下关键信息：

- **功能名称**：用户如何称呼这个功能
- **核心价值**：一句话说清"这能帮用户省什么事/解决什么痛点"
- **目标用户**：谁会使用这个功能
- **前置条件**：使用该功能需要满足的条件（权限、配置等），以及如何获取权限
- **功能入口**：用户从哪里进入该功能
- **操作流程**：用户需要执行哪些步骤
- **状态变化**：操作过程中/完成后界面有什么变化
- **限制与规则**：常见的限制条件、业务规则
- **异常处理**：用户可能遇到的问题及解决方案

### 3. 查找原型文件（用于截图）

在 PRD 文档同目录或子目录中查找原型文件：

1. 查找 `原型.html` 或 `原型/` 目录下的 HTML 文件
2. 如果找到原型 HTML 文件，进入截图流程（步骤 4）
3. 如果没有原型文件，跳过截图，docx 中用图片占位符代替

### 4. 截图原型页面（如有原型 HTML）

调用 `scripts/screenshot.py` 对原型 HTML 进行无头浏览器截图：

```bash
python3 $SKILL_DIR/scripts/screenshot.py <截图配置.json>
```

**准备工作：** 生成截图配置 JSON，描述需要截图的页面和点击步骤。配置格式见 `scripts/screenshot_schema_example.json`。

**截图配置 JSON 核心字段：**

```json
{
  "html_path": "原型HTML的绝对路径",
  "output_dir": "/tmp/guide-screenshots",
  "viewport": {"width": 1440, "height": 900},
  "pages": [
    {
      "name": "页面标识",
      "file": "输出文件名.png",
      "clicks": [
        {"type": "css",  "selector": "CSS选择器", "wait_after_ms": 800},
        {"type": "text", "selector": "按钮文本",  "wait_after_ms": 800},
        {"type": "eval", "selector": "JS代码",    "wait_after_ms": 500}
      ],
      "full_page": true
    }
  ]
}
```

**截图要点：**
- 设计尺寸 1440x900（B端规范）
- 每个页面/弹窗截一张图，文件名语义化
- 截图前等待页面渲染完成（500-800ms）
- 使用 `full_page=True` 截取完整页面
- 点击支持三种方式：CSS 选择器、文本匹配、JS eval

**环境依赖：**

```bash
pip3 install playwright
playwright install chromium
```

### 5. 生成用户手册.md

按照以下章节结构生成完整的 Markdown 用户手册：

```markdown
# [功能名称] - 使用指南

## 功能概述

[第一句话点明核心价值：这能帮用户省什么事/解决什么痛点]

[第二句话补充说明：适用场景、核心能力]

## 使用步骤

### 权限与准备

[如有权限限制，用引用块说明]

> 💡 **权限说明**：本功能目前仅面向 [用户类型] 开放。
> 
> **如何获取权限**：[提供具体路径，如联系谁、点击哪里申请]

[其他前置条件，如配置要求、数据准备等]

### 操作指引

#### 一、[第一个任务模块]

[用流畅的叙述描述操作流程，不要用"操作：... 结果：..."格式]

1. **[步骤名称]**
   [用自然语言描述：点击哪里、看到什么、做什么]
   
   *[此处建议插入图片：XXX界面截图]*
   
   > 💡 **小贴士**：[如有注意事项或技巧，用引用块补充]

2. **[步骤名称]**
   [继续描述...]

#### 二、[第二个任务模块]

...

## 常见问题

> ❓ **问题：[用户可能遇到的问题]**
> 
> **解决方法**：[具体解决方案]

## 注意事项

> ⚠️ **[注意事项标题]**：[具体说明]
> 
> [如有需要，补充更多细节]

> 📌 **[重要提示]**：[需要用户特别注意的事项]

## 相关说明

[如有需要，补充业务术语解释、状态说明表格、相关功能链接等]
```

### 6. 生成用户手册.docx

调用 `scripts/gen_docx.py` 生成 Word 文档，**将截图嵌入对应章节**：

```bash
python3 $SKILL_DIR/scripts/gen_docx.py <docx配置.json> [输出路径.docx]
```

**准备工作：** 生成 docx 配置 JSON，描述封面信息和正文内容结构。配置格式见 `scripts/doc_schema_example.json`。

**docx 配置 JSON 核心结构：**

```json
{
  "cover": {
    "title": "产品名称",
    "subtitle": "使用指南",
    "version": "V1.0",
    "date": "2026-06-26",
    "author": "作者"
  },
  "output_dir": "输出目录（PRD同目录）",
  "doc_filename": "用户手册.docx",
  "screenshot_dir": "/tmp/guide-screenshots",
  "content": [
    {"type": "heading", "level": 1, "text": "功能概述"},
    {"type": "paragraph", "text": "正文段落..."},
    {"type": "quote", "prefix": "小贴士:", "text": "提示内容..."},
    {"type": "list", "bullets": ["条目1", "条目2"]},
    {"type": "screenshot", "file": "01_dashboard.png", "caption": "图1: 监控大盘"},
    {"type": "table", "headers": ["列A","列B"], "rows": [["值1","值2"]]},
    {"type": "page_break"}
  ]
}
```

**支持的 content 类型：** heading（标题）、paragraph（段落）、quote（引用块）、list（列表）、table（表格）、screenshot（截图+图注）、page_break（分页）、blank（空行）

**docx 生成要点：**
- 封面：标题 + 副标题 + 版本 + 日期 + 作者
- 正文：与 `用户手册.md` 相同的章节结构
- 图片：screenshot 类型自动插入截图，居中显示，下方添加灰色图注
- 如果没有截图（无原型文件），自动显示文字占位符
- 表格使用 `Table Grid` 样式，表头加粗

### 7. 生成功能简介.md

独立的功能简介文档，按照以下结构输出：

```markdown
# 功能简介

### 模块名称：[功能名称]
- **关键词**：[5-10个关键词，逗号分隔]
- **适用场景**：当用户[需要做什么]时，使用本模块[完成什么]
- **核心摘要**：[一段话概括核心能力和流程]
- **输入与产出**：输入[什么]，产出[什么]
- **简单用法**：1. [步骤1] -> 2. [步骤2] -> 3. [步骤3] -> ...
- **访问链接**：[URL或入口路径]
- **前置条件**：[权限、配置等条件]
- **常见问题(FAQ)**：
  - Q: [问题1]
    A: [答案1]
  - Q: [问题2]
    A: [答案2]
- **操作手册链接**：[指向用户手册的路径或链接]
```

## 写作规范

### 语言风格：去"测试用例"化

❌ **避免**（测试用例风格）：
```
操作：点击「申请退款」按钮
结果：弹出退款申请弹窗
```

✅ **推荐**（任务导向叙述）：
```
在订单详情页点击「申请退款」按钮，系统会弹出退款申请弹窗供您填写。
```

### 视觉元素使用

| 元素 | 使用场景 | 示例 |
|------|----------|------|
| 💡 | 小贴士、技巧提示 | `> 💡 **小贴士**：安装后的技能可在「我的技能」中找到` |
| ⚠️ | 警告、限制说明 | `> ⚠️ **注意**：审核中的技能无法直接卸载` |
| 📌 | 重要提示、强调 | `> 📌 **重要**：同一场景只能启用一个审批流` |
| ❓ | 常见问题 | `> ❓ **问题：找不到某个功能？**` |
| 引用块 `>` | 提示、警告、FAQ | 见上方示例 |
| 表格 | 状态说明、对比说明 | 状态机、字段说明等 |

### 前置条件：提供解决路径

❌ **避免**（冷冰冰的阻断）：
```
- 您需要是 ERP 用户才能使用技能管理功能
- 非ERP用户将看不到技能tab
```

✅ **推荐**（增强服务意识）：
```
> 💡 **权限说明**：技能管理功能目前仅面向 **ERP 用户** 开放。
> 
> **如何获取权限**：如果您登录后未在顶部导航栏看到「技能」Tab，说明暂无权限。请联系系统管理员申请开通 ERP 账号。
```

### 功能概述：体现业务价值

❌ **避免**（干瘪）：
```
技能管理中心是 Zero 平台的统一技能管理入口，帮助您发现、安装和管理各类技能。
```

✅ **推荐**（价值优先）：
```
技能管理中心是 Zero 平台的统一效能工具库。无论您是需要**开发排错、数据分析还是内容创作**，都能在这里找到开箱即用的 AI 技能，一键安装即可使用，快速提升工作效率。
```

## 输出检查清单

生成完成后，检查以下要点：

- [ ] **三个文件已生成**：用户手册.docx、用户手册.md、功能简介.md
- [ ] **文件位置正确**：三个文件都在 PRD 文档同目录下
- [ ] **docx 包含截图**：如原型 HTML 存在，docx 中已嵌入对应页面截图
- [ ] **功能概述**是否第一句话就点明了"解决什么痛点"
- [ ] **前置条件**是否提供了"如何获取权限"的解决路径
- [ ] **步骤描述**是否采用了流畅的任务导向叙述（而非"操作：... 结果：..."）
- [ ] **md 文件**是否在关键步骤后添加了图片占位符
- [ ] **emoji 图标**是否使用了 💡 ⚠️ 📌 ❓ 增强视觉引导
- [ ] **引用块** `>` 是否突出了重要提示和常见问题
- [ ] **功能简介.md** 是否包含完整的模块信息（关键词/场景/摘要/用法/FAQ）
- [ ] **语言**是否避免了技术术语、具有"对话感"

## 脚本工具

本技能在 `scripts/` 目录下提供两个参数化脚本，AI 运行时生成 JSON 配置后调用：

| 脚本 | 功能 | 用法 |
|------|------|------|
| `scripts/screenshot.py` | 原型 HTML 无头浏览器截图 | `python3 screenshot.py <截图配置.json>` |
| `scripts/gen_docx.py` | 根据配置 JSON 生成 docx 文档 | `python3 gen_docx.py <docx配置.json> [输出路径]` |

**工作流程：**
1. AI 分析 PRD 内容，生成截图配置 JSON → 调用 `screenshot.py` 截图
2. AI 根据用户手册内容，生成 docx 配置 JSON → 调用 `gen_docx.py` 生成 Word
3. 配置 JSON 模板见 `scripts/doc_schema_example.json` 和 `scripts/screenshot_schema_example.json`

> 💡 `$SKILL_DIR` 指向技能目录（如 `~/.claude/skills/user-guide-creator`），AI 运行时替换为实际路径。

## 环境依赖

运行本技能需要以下 Python 包：

```bash
pip3 install playwright python-docx
playwright install chromium
```

如环境缺少依赖，docx 中的截图部分用文字占位符代替，不影响其他文件生成。
